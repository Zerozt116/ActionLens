**论文标题：** ST-HOI: A Spatial-Temporal Baseline for Human-Object Interaction Detection in Videos

**来源：** ICDAR '21 Workshop (2021) / arXiv:2105.11731

**作者：** Meng-Jiun Chiou, Chun-Yu Liao, Li-Wei Wang, Roger Zimmermann, Jiashi Feng (NUS & ASUS)

**论文地址：** https://arxiv.org/abs/2105.11731

**代码地址：** https://github.com/coldmanck/VidHOI

## 一、论文解决什么问题

传统的人-物交互（HOI）检测方法大多针对静态图像设计，只能识别单帧中的空间关系（如"坐在椅子上"），却无法区分需要时间信息才能判断的时序交互（如"推门"与"拉门"、"靠近"与"远离"）。将这些 2D 方法直接逐帧应用于视频时，它们把每帧当作独立同分布数据，无法利用邻帧间的时序上下文。

一个直觉性的想法是：将 2D 骨干网络替换为 3D-CNN（如 SlowFast），利用 3D 卷积编码时间信息。然而论文通过实验揭示了一个**反直觉的现象**：简单替换 2D 骨干为 3D 骨干后，VideoHOI 性能仅提升约 2%，远不如预期。更严重的是，这一现象在时空动作检测（STAD）和时空场景图生成（STSGG）中也有观察到，3D 骨干甚至有损性能。

论文的核心贡献在于找到了这个现象的根源——**特征不一致（Feature Inconsistency）问题**，并提出了针对性的解决方案。

---

## 二、核心发现：特征不一致问题

### 2.1 问题根源

常规 STAD 方法的标准做法是：先用 3D-CNN 提取时空特征图（形状 d×T×H×W），然后沿时间轴做平均池化（temporal pooling）得到 d×1×H×W，再用 RoI Pooling 裁剪目标区域特征。

**这等价于：在目标关键帧的 RoI 位置，跨整个视频片段裁剪特征——完全忽略了人和物体的运动。**

对于运动中的目标，其在邻帧中的位置很可能与关键帧中的位置不同。在关键帧位置做 temporal-RoI pooling，可能采到的是其他人/物体或者无意义的背景。这就是"特征不一致"问题。

```
关键帧中人的位置:  (x1, y1)
第 t-2 帧中人的位置: (x2, y2)  ← 人已移动
第 t-1 帧中人的位置: (x3, y3)  ← 人继续移动
关键帧中人的位置:  (x1, y1)

传统方法在 (x1, y1) 处对所有帧做 RoI Pooling:
  → 第 t-2, t-1 帧采到的可能是背景或其他物体
  → 特征不一致，性能不升反降
```

### 2.2 初步验证

论文在三个任务上验证了"2D→3D 骨干替换"的效果：

| 任务 | 替换后相对变化 |
|------|---------------|
| VideoHOI（triplet mAP） | 仅 +2% |
| STAD（AVA, mAP） | -5%~-10%（有害） |
| STSGG（Action Genome, R@20） | -5%~-10%（有害） |

而加上轨迹特征后，VideoHOI 的相对提升从 2% 跃升至 **23%**，验证了"正确定位"的时序信息的重要性。

## 三、ST-HOI 方法设计

### 3.1 整体架构

ST-HOI 以关键帧为中心，输入 T 帧视频片段，使用 SlowFast 作为 3D 骨干，利用 N 条目标轨迹（含 M 条人体轨迹）提取三种时序特征，最后拼接分类：

```
输入视频片段 (T帧, 以关键帧为中心)
       ↓
  SlowFast 3D 骨干 → 时空特征图 (d×T×H×W)
       ↓
  ┌──────────────────────────────────────────┐
  │  三种时序特征提取                          │
  │                                           │
  │  (a) 轨迹特征 j: 逐帧 bbox 坐标           │
  │  (b) 正确定位视觉特征 v̄: ToI Pooling       │
  │  (c) 时空掩码姿态特征 p̄: 骨骼+空间掩码     │
  └──────────────────────────────────────────┘
       ↓
  拼接: v_so = [v̄_s; v̄_u; v̄_o; j_s; j_o; p̄_so]
       ↓
  两层全连接 → 50 类交互 sigmoid 分类
```

### 3.2 正确定位视觉特征（ToI Pooling）

核心思想：**反转 temporal pooling 和 RoI pooling 的顺序**。

传统做法：先时间池化 → 再 RoI 池化（错误，位置不对齐）。
ST-HOI 做法：先逐帧 RoI 池化（根据每帧的实际 bbox）→ 再时间平均池化。

公式表示为：

v̄ᵢ = (1/T) Σ RoIAlign(vₜ, jᵢ,ₜ)

其中 jᵢ,ₜ 是目标 i 在第 t 帧的实际 bounding box。这确保了每帧都从目标的**真实位置**裁剪特征，解决了特征不一致问题。该方法也被称为 Tube-of-Interest (ToI) Pooling。

对于人-物交互，还利用**联合包围框**（union bbox）提取上下文视觉特征（如"放风筝"时人和风筝之间的天空）。

### 3.3 时空掩码姿态特征

结合人体骨骼和空间位置信息，并扩展到时间维度：

**每帧生成：**
1. **骨骼掩码**：用 FastPose 检测 17 个关键点，连成骨架画在 H×W 的二值掩码上，每条连线有不同灰度值 x∈[0,1]。
2. **空间掩码**：为人和物体各生成一个二值掩码（bbox 内为 1，外为 0），共 2 通道。

**拼接后**得到 3×H×W 的时空掩码姿态特征 pᵢ,ₜ = [sᵢ,ₜ; h̄ᵢ,ₜ]。

**时序建模**：将 T 帧的掩码特征送入两个 3D 卷积层，再做空间最大池化和时间平均池化，展平后得到最终的时空掩码姿态特征 p̄ᵢ,ₜ。

这一模块的创新在于：不仅考虑单帧的姿态和空间关系，还建模了**姿态随时间的变化模式**（如骑马时人和马应该同步移动）。

### 3.4 轨迹特征

直接将人和物体的逐帧 bounding box 坐标 jᵢ ∈ R^(T×4) 作为特征，编码目标的运动轨迹信息。如果某帧中 bbox 不可用（轨迹不连续），则用整幅图像作为默认框。

### 3.5 预测与训练

将上述三种特征拼接后送入两层全连接网络，使用逐类二元交叉熵损失训练（VideoHOI 是多标签分类任务）。推理时按 softmax 分数排序，仅评估 top-100 预测。

## 四、VidHOI 基准数据集

论文从 VidOR 数据集构建了 VidHOI 视频 HOI 基准，填补了该领域的空白：

| 数据集 | 视频 | 定位目标 | 视频时长 | 标注帧数 | 物体类别 | 谓词类别 | HOI 类别 | HOI 实例数 |
|--------|------|---------|---------|---------|---------|---------|---------|-----------|
| HICO-DET | 否 | 是 | - | 47K | 80 | 117 | 600 | 150K |
| V-COCO | 否 | 是 | - | 10K | 80 | 25 | 259 | 16K |
| AVA | 是 | 否 | 108h | 3.7M | - | 49 | 80 | 1.6M |
| CAD-120 | 是 | 是 | 0.57h | 0.5K | 13 | 6 | 10 | 32K |
| Action Genome | 是 | 部分 | 82h | 234K | 35 | 25 | 157 | 1.7M |
| **VidHOI** | **是** | **是** | **70h** | **7,122** | **78** | **50** | **557** | **755K** |

关键特点：

- **50 个谓词**：包括动作（push, pull, lift 等）和空间关系（next_to, behind 等），其中 25 个为时序相关谓词，但仅占数据集约 5%。
- **关键帧中心策略**：以 1Hz 采样关键帧，标注在 t±0.5s 内有效（与 AVA 协议对齐）。
- **训练集**：193,911 关键帧 / 6,366 视频；**验证集**：22,808 关键帧 / 756 视频。
- **评估指标**：mAP，基于 557 个三元组（人-谓词-物），需同时满足人/物 bbox IoU>0.5 且交互正确。

## 五、实验与结果

### 5.1 主要实验结果（VidHOI）

| 模型 | Full mAP | Non-rare | Rare | 相对提升 |
|------|----------|----------|------|---------|
| **Oracle 模式（使用 GT 轨迹）** |
| 2D 模型 | 14.1 | 22.9 | 11.3 | - |
| 3D 模型（SlowFast 骨干） | 14.4 | 23.0 | 12.6 | +2.1% |
| Ours-T（仅轨迹） | 17.3 | 26.9 | 16.8 | +22.7% |
| Ours-T+V（轨迹+视觉） | 17.3 | 26.9 | 16.3 | +22.7% |
| Ours-T+P（轨迹+姿态） | 17.4 | 27.1 | 16.4 | +23.4% |
| Ours-T+V+P（完整） | **17.6** | **27.2** | **17.3** | **+24.8%** |
| **Detection 模式（使用预测轨迹）** |
| 2D 模型 | 2.6 | 4.7 | 1.7 | - |
| 3D 模型 | 2.6 | 4.9 | 1.9 | 0.0% |
| Ours-T+V+P（完整） | **3.1** | **5.9** | **2.1** | **+19.2%** |

### 5.2 时序谓词 vs 空间谓词

这是论文最具说服力的实验之一：

| 模型 | 时序 mAP | 时序相对提升 | 空间 mAP | 空间相对提升 |
|------|---------|------------|---------|------------|
| 2D 模型 | 8.3 | - | 18.6 | - |
| 3D 模型 | 7.7 | **-7.2%** | 20.9 | +12.3% |
| Ours-T | 14.4 | **+73.5%** | 24.7 | +32.8% |
| Ours-T+V+P | **14.4** | **+73.5%** | **25.0** | **+34.4%** |

3D 模型在时序谓词上**性能反而下降 7.2%**（因为特征不一致），而 ST-HOI 将时序 mAP 从 8.3 提升至 14.4（+73.5%），空间 mAP 也有显著提升。

### 5.3 关键观察

1. **轨迹特征是最重要的贡献者**：仅加轨迹（Ours-T）就带来绝大部分提升。
2. **视觉特征 V 和姿态特征 P 是互补的**：完整模型在所有设置下均最优。
3. **Oracle 与 Detection 差距大**：说明轨迹生成质量是主要瓶颈，这也是未来工作的改进方向。
4. **长尾效应明显**：Full mAP 接近 Rare mAP，说明多数 HOI 类别样本稀少。
5. **不同特征对不同谓词的贡献不均等**：如 behind、beneath 等空间谓词受益于完整模型，而 watch、ride 等在子模型上表现更好，这与样本数量有关。

## 六、关键洞见总结

1. **特征不一致是 3D 骨干在 HOI 中失效的根本原因**：temporal pooling + RoI pooling 的顺序对移动目标会产生错位特征，这是一个影响广泛的架构设计陷阱。
2. **ToI Pooling 是简单而有效的解决方案**：反转池化顺序，先逐帧 RoI 再时间平均，即可恢复正确定位的时序信息。
3. **轨迹特征比视觉特征更重要**：在 VideoHOI 中，知道"目标在哪里移动"比"每帧看到什么"更有区分力。
4. **时序信息对空间谓词同样有帮助**：不仅时序谓词（push/pull）受益，空间谓词（next_to/behind）也从轨迹特征中获得了 +32.8% 提升。
5. **VideoHOI 仍有很大的提升空间**：Detection 模式下 mAP 仅 3.1，说明轨迹生成质量和模型整体能力距离实用还有距离。

## 七、与本课题的关联

ST-HOI 与本课题中"人-物交互检测"模块（课题方案 §4.4）直接相关，为理解视频中人与物体的交互关系提供了重要的方法论基础。

### 7.1 人-物交互在课题中的角色

课题目标是输出"谁、在什么时间、做了什么"的结构化行为时间线。其中"做了什么"不仅包含人的动作（如喝水），还包含人与物体的交互关系（如"使用打印机"、"拿起杯子"）。ST-HOI 正是解决后者的方法。

### 7.2 与 SlowFast 骨干的配合

ST-HOI 直接使用 SlowFast 作为 3D 骨干网络。这意味着课题中可以统一使用 SlowFast 做视频特征提取：

```
视频片段 → SlowFast 骨干（提取时空特征图）
              ↓
    ┌────────┴────────┐
    ↓                 ↓
  动作分类           HOI 检测
  (SlowFast论文)     (ST-HOI方法)
    ↓                 ↓
  "在喝水"          "人-使用-杯子"
```

### 7.3 轨迹特征与 ByteTrack 的衔接

ST-HOI 严重依赖目标轨迹作为输入特征，而课题中 ByteTrack 正是负责多目标跟踪的模块。这形成了自然的衔接：

```
YOLO 检测 → ByteTrack 跟踪 → 生成人/物轨迹
                                   ↓
                        ST-HOI 利用轨迹做 HOI 检测
```

ByteTrack 输出的逐帧 bounding box 可以直接作为 ST-HOI 的轨迹输入 jᵢ ∈ R^(T×4)。

### 7.4 ToI Pooling 对课题的启示

ToI Pooling 的核心思想——根据目标轨迹逐帧定位后再池化——对课题中所有需要从视频中提取实例特征的场景都有参考价值：不仅 HOI 检测需要，动作分类、行为时间定位等模块在处理运动目标时都应考虑类似的正确定位策略。

### 7.5 VidHOI 数据集的参考意义

VidHOI 的 50 个谓词中包含了课题关注的典型交互类型（如 hold、push、pull、watch、sit_on 等），其关键帧中心评估策略与 AVA 的 1Hz 标注协议一致，可以方便地与课题中其他模块的评估对齐。

## 八、关键 Takeaways

1. **不要盲目升级 2D→3D**：在涉及实例级特征的任务中，简单替换 2D 骨干为 3D 骨干可能无效甚至有害，需要先分析特征池化是否正确对齐。
2. **ToI Pooling 是视频 HOI 的正确做法**：先逐帧 RoI 池化（利用轨迹）→ 再时间平均池化。
3. **轨迹是视频 HOI 最重要的特征**：仅轨迹特征就能带来 +23% 的相对提升，远超 3D 骨干本身的 +2%。
4. **空间和时间特征高度互补**：完整模型（轨迹+视觉+姿态）在所有评估中均最优。
5. **VideoHOI 仍处于早期阶段**：Detection 模式下 mAP 仅 3.1，说明轨迹生成、特征提取、交互分类三个环节都有很大提升空间。
